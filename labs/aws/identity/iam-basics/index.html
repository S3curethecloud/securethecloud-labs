AWS IAM Basics

Cloud: AWS Domain: Identity Level: Foundation Estimated Time: 45 minutes

A foundational lab focused on how AWS Identity and Access Management (IAM) governs every action in the cloud. This lab emphasizes reasoning, not console memorization.

Why IAM Matters

IAM is the security control plane of AWS. Every API call, service interaction, and workload execution is evaluated through IAM.

  • Most AWS breaches involve IAM misconfiguration
  • Over-permissioned roles silently expand blast radius
  • Identity mistakes compound across services
Security Reality: IAM failures rarely cause immediate outages — they create long-lived exposure that attackers exploit later.

Core IAM Concepts

Principals

A principal is any entity that can make a request to AWS: users, roles, or AWS services.

Policies

Policies define what actions are allowed. They do not grant access by themselves — they must be attached to a principal.

Evaluation Logic

  • Explicit Deny always wins
  • Implicit deny is the default
  • All applicable policies are evaluated together
Architecture Insight: IAM is evaluated on every request, not at login. Temporary credentials (STS) are the foundation of modern AWS security.

Lab Objectives

  • Differentiate IAM users vs roles
  • Understand trust vs permission policies
  • Reason about least-privilege design
  • Explain IAM decisions clearly in interviews

Interview Focus

When discussing IAM in interviews, emphasize why roles are preferred over long-lived users, and how identity design limits blast radius.

Interview Tip: If you say “everything goes through IAM,” be prepared to explain policy evaluation order and STS.

What This Lab Is Not

  • A console click-through tutorial
  • A certification cram sheet
  • A least-privilege checkbox exercise

This lab is about thinking like a cloud security architect.
