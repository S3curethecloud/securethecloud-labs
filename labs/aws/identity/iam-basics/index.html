AWS IAM Basics

This lab introduces AWS Identity and Access Management (IAM) using production-grade patterns. The focus is not on clicking through the console, but on understanding how identity decisions shape security, blast radius, and operational control.

Why This Lab Matters

IAM is the security control plane for AWS. Every API call, workload, and human action is evaluated through IAM.

  • Misconfigured IAM is the #1 cause of cloud breaches
  • Over-permissioned roles create silent blast radius
  • Identity mistakes compound across services

Core IAM Concepts

Principals

A principal is an entity that can make a request to AWS. This includes users, roles, and services.

Policies

Policies define what is allowed. They do not grant access by themselves — they must be attached to a principal.

Evaluation Logic

  • Explicit Deny always wins
  • Default is implicit deny
  • Multiple policies are evaluated together
Architecture Insight: IAM is evaluated on every request, not at login. Temporary credentials (STS) are the foundation of modern AWS security.

Lab Objectives

  • Understand IAM users vs roles
  • Read and reason about IAM policies
  • Identify over-permissioned access
  • Explain IAM decisions in an interview setting

What This Lab Is Not

  • A console click-through guide
  • A certification cram sheet
  • A least-privilege checkbox exercise

This lab is about thinking like a cloud security architect.
