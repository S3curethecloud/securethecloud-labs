GCP IAM Basics

Cloud: GCP Domain: Identity Level: Foundation Estimated Time: 45 minutes

A foundational lab focused on how Google Cloud Identity and Access Management (IAM) governs every action in GCP. This lab emphasizes reasoning, not console memorization.

Why IAM Matters

IAM is the security control plane of GCP. Every API call, service interaction, and workload execution is evaluated through IAM.

  • Most GCP breaches involve IAM misconfiguration
  • Over-permissioned IAM roles and bindings silently expand blast radius
  • Identity mistakes compound across services
Security Reality: IAM failures rarely cause immediate outages — they create long-lived exposure that attackers exploit later.

Core IAM Concepts

Principals

A principal is any entity that can make a request to GCP: users, groups, service accounts, or Google-managed services.

Policies

Policies define what actions are allowed. In GCP, access is granted through IAM roles and bindings between principals and resources.

Evaluation Logic

  • Explicit Deny always wins
  • Implicit deny is the default
  • All applicable IAM roles / bindings are evaluated together
Architecture Insight: IAM is evaluated on every request, not at login. Short-lived service account credentials are the foundation of modern GCP security.

Lab Objectives

  • Understand GCP IAM principals and service accounts
  • Reason about IAM roles and bindings
  • Identify over-permissioned access
  • Explain GCP IAM decisions clearly in interviews

Interview Focus

When discussing GCP IAM in interviews, emphasize why service accounts and short-lived credentials are preferred and how IAM bindings limit blast radius.

Interview Tip: If you say “everything goes through IAM,” be prepared to explain role inheritance, resource hierarchy, and credential lifetimes.

What This Lab Is Not

  • A console click-through tutorial
  • A certification cram sheet
  • A least-privilege checkbox exercise

This lab is about thinking like a cloud security architect.
