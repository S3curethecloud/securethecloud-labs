📖 Concept Deep Dives

1. What is data exfiltration scenario design?

Data exfiltration scenario design is the safe planning of tests that evaluate whether an AI workflow could expose sensitive information. The scenario should use synthetic data classes and bounded evidence — not real customer records or secrets.

2. Why must sensitive data be represented synthetically?

Using synthetic data keeps the learning exercise reviewer-safe. It allows the scenario to model exposure risk without handling real:

  • Customer records or PII
  • Credentials or API keys
  • Regulated data (PHI, PCI, GDPR)
  • Confidential business information
  • Trade secrets or intellectual property

3. How do data boundaries and authority boundaries differ?

Data boundary — What information may be accessed or disclosed?

  • Tenant scope (Customer A vs. Customer B)
  • Sensitivity level (Public vs. Confidential)
  • Data classification (PII, Financial, PHI)

Authority boundary — Who or what is allowed to request, transform, or release that information?

  • Identity and role verification
  • Policy approval requirements
  • Tool permission checks

A safe scenario reviews both boundaries.

4. What controls should a data exposure scenario test?

Data exposure scenarios should map to specific controls:

  • Data classification — Sensitivity labels on information
  • Tenant scoping — Access limited to authorized tenants
  • Least privilege — Minimal necessary access
  • Retrieval filtering — Only authorized sources retrieved
  • Tool permission checks — Authorization before data access
  • Output policy — Redaction or blocking of sensitive data
  • Human approval — Escalation for sensitive disclosures
  • Logging — All data access attempts recorded
  • Fail-closed behavior — Deny access when uncertain
5. How should impact and uncertainty be recorded?

Impact description should answer:

  • What type of data could be exposed? (Synthetic data class)
  • Why does this matter? (Regulatory, contractual, competitive risk)
  • What systems or customers could be affected?

Uncertainty should describe:

  • What was not tested in this scenario
  • What evidence is synthetic vs. real
  • What cannot be concluded from the scenario
  • What follow-up testing would increase confidence
6. How should data exfiltration findings be documented safely?

A safe finding should include:

  • Scenario objective — What disclosure risk was evaluated?
  • Synthetic data class — What type of data was modeled?
  • Expected control — What should have prevented exposure?
  • Observed behavior — What actually happened?
  • Uncertainty — What remains unknown?
  • Impact — Why does this matter?
  • Remediation — What should be fixed?

Avoid: Exposing real data, providing operational exfiltration steps, or handling real credentials.

⚠️ High-Risk Anti-Pattern

A dangerous pattern is using real customer data, credentials, secrets, or production outputs to prove that a data exposure scenario is "realistic" or "valid."

❌ Unsafe Pattern

  • Real customer data
  • Live secrets or tokens
  • Credential material
  • Production system interaction
  • Live data movement
  • Unsupported breach claims

✅ Safe Alternative

  • Synthetic data class
  • Bounded scenario
  • No live data movement
  • No credentials or secrets
  • Control-mapped evidence
  • Remediation-focused finding
