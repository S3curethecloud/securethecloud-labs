📚 Concept Deep Dives

1. What is AI red-team scenario design?

AI red-team scenario design is the structured planning of safe tests that explore how an AI system might fail. The goal is not to attack real systems, but to define a bounded scenario, expected controls, evidence requirements, and safe reporting language.

2. Why must red-team work be scoped and authorized?

Red-team work can cross sensitive boundaries if not properly scoped. A safe scenario identifies: what is in scope, what is out of scope, who authorized the review, what data is synthetic, and which actions are forbidden.

3. What belongs in a safe AI red-team scenario?

A safe scenario includes: objective, authorization boundary, system context, failure mode, expected control, synthetic inputs, observation steps, evidence template, uncertainty notes, and remediation guidance.

4. How do failure modes map to controls?

Each failure mode should map to a control:

  • Prompt injection → instruction hierarchy and untrusted-input handling
  • Tool abuse → permissions and approval gates
  • Retrieval poisoning → source authority, tenant scope, and context validation
5. How should evidence and uncertainty be recorded?

Evidence should distinguish: what was observed, what was expected, what remains unknown, and what cannot be concluded. Good red-team evidence avoids unsupported compromise claims and uses bounded risk language.

6. What makes a red-team finding portfolio-ready?

A portfolio-ready finding explains the scenario, control objective, evidence, impact, uncertainty, and remediation in a way that a reviewer can evaluate without needing access to private systems or sensitive data.

⚠️ High-Risk Anti-Pattern

A dangerous red-team pattern is attempting live attacks, using real sensitive data, or testing outside authorization while describing the activity as "learning" or "research."

❌ Unsafe Pattern

  • Unclear authorization
  • Real customer data
  • Live credential material
  • Production system interaction
  • Exploit execution
  • Unsupported compromise claims

✅ Safe Alternative

  • Synthetic data
  • Bounded scenario
  • No live execution
  • Documented scope
  • Reviewer-safe evidence
  • Remediation-focused finding
