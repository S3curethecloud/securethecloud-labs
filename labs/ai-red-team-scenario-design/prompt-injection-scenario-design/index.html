📖 Concept Deep Dives

1. What is prompt-injection scenario design?

Prompt-injection scenario design is the safe planning of tests that evaluate whether an AI system can distinguish trusted instructions from untrusted user input, retrieved content, tool output, or other external text. The goal is to assess controls — not to exploit systems.

2. Why is instruction hierarchy important?

Instruction hierarchy defines which instructions have authority. A safe scenario checks whether system and policy instructions remain authoritative when lower-trust content attempts to influence behavior. Without this, prompt injection becomes operational risk.

3. Where does untrusted input enter an AI workflow?

Untrusted input can come from multiple sources:

  • User messages and prompts
  • Uploaded documents and files
  • Tickets and support requests
  • Retrieved webpages and search results
  • Tool output and API responses
  • Email content and chat history
  • Logs and audit trails

A scenario should identify the input source and trust level before testing controls.

4. How does retrieved content become a prompt-injection risk?

Retrieved content becomes risky when the model treats external text as authority instead of context. A safe scenario checks whether the system validates:

  • Source authority (policy document vs. vendor note)
  • Tenant scope (customer A vs. customer B)
  • Sensitivity classification (public vs. confidential)
  • Freshness and relevance
5. What controls should a scenario test?

Prompt-injection scenarios should map to specific controls:

  • Instruction hierarchy — Trusted vs. untrusted content separation
  • Source trust labeling — Marking user input as untrusted
  • Retrieval filtering — Scope and authority checks
  • Tool permission checks — Authorization before action
  • Refusal behavior — Safe "cannot do that" responses
  • Human approval gates — Escalation for sensitive actions
  • Evidence capture — Logging decisions and reasoning
6. How should prompt-injection findings be documented safely?

A safe finding should include:

  • Objective — What was being evaluated?
  • Scope — What was in/out of bounds?
  • Untrusted input source — Where did the test input come from?
  • Expected control — What should have prevented the issue?
  • Observed behavior — What actually happened?
  • Uncertainty — What remains unknown?
  • Impact — Why does this matter?
  • Remediation — What should be fixed?

Avoid: Publishing reusable jailbreak instructions or operational payloads.

⚠️ High-Risk Anti-Pattern

A dangerous pattern is publishing or running reusable prompt-injection payloads against real systems while calling the activity "training," "testing," or "research."

❌ Unsafe Pattern

  • Unclear authorization
  • Live target system
  • Reusable jailbreak text
  • Real sensitive data
  • Real tool execution
  • Unsupported compromise claims

✅ Safe Alternative

  • Authorized scope
  • Synthetic scenario description
  • No operational payload library
  • No live execution
  • Control-mapped evidence
  • Remediation-focused finding
