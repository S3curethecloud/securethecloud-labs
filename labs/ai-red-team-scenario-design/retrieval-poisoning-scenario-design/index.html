📖 Concept Deep Dives

1. What is retrieval poisoning scenario design?

Retrieval poisoning scenario design is the safe planning of tests that evaluate whether an AI workflow can be influenced by untrusted, stale, low-authority, tenant-crossing, or misleading retrieved content. The goal is to assess retrieval controls — not to poison real data sources.

2. Why does source authority matter?

Source authority determines whether retrieved content should be treated as:

  • Trusted evidence — Approved policy documents, system-of-record
  • Low-confidence context — User notes, vendor emails, support tickets
  • Untrusted input — External web content, unverified sources

A safe scenario identifies which sources have authority and which do not, then tests whether the system respects those boundaries.

3. How do stale or low-authority sources create risk?

📊 Risk Scenarios

  • Stale policy document — Agent follows outdated compliance rules
  • Vendor note injection — External text overrides system instructions
  • Support ticket manipulation — Attacker-controlled content poisons context
  • Outdated pricing data — Agent recommends incorrect financial actions

A scenario should define freshness expectations and review how the system handles weak, stale, or conflicting evidence.

4. Where can tenant or context boundaries fail?

Boundary failures can occur when retrieved content crosses:

  • Tenant boundaries — Customer A's documents leak to Customer B
  • Project boundaries — Internal project context visible to unauthorized users
  • Role boundaries — Regular user sees admin-only documents
  • Environment boundaries — Production receives development/test data
  • Sensitivity boundaries — Confidential or regulated data exposed
5. What controls should a retrieval-risk scenario test?

Retrieval-risk scenarios should map to specific controls:

  • Source allowlists — Only approved sources can be retrieved
  • Source ranking — Higher-authority sources outrank lower ones
  • Freshness checks — Stale content is flagged or excluded
  • Tenant scoping — Retrieval respects tenant boundaries
  • Context labeling — Retrieved content marked as context, not authority
  • Citation requirements — Claims must be supported by evidence
  • Fail-closed behavior — Low confidence leads to escalation, not unsafe action
6. How should retrieval-risk findings be documented safely?

A safe finding should include:

  • Retrieval source — What source was tested?
  • Authority level — Trusted, low-confidence, or untrusted?
  • Expected control — What should have prevented the issue?
  • Observed behavior — What actually happened?
  • Uncertainty — What remains unknown?
  • Risk impact — Why does this matter?
  • Remediation — What should be fixed?

Avoid: Modifying live indexes, tampering with real sources, or publishing source-poisoning steps.

⚠️ High-Risk Anti-Pattern

A dangerous pattern is modifying or seeding real retrieval sources to influence model behavior while treating the activity as "safe testing" or "research."

❌ Unsafe Pattern

  • Real corpus mutation
  • Vector database writes
  • Source tampering
  • Customer data exposure
  • Tenant boundary crossing
  • Unsupported compromise claims

✅ Safe Alternative

  • Synthetic evidence artifact
  • Documented source authority
  • No live index mutation
  • No customer data access
  • Control-mapped observation
  • Remediation-focused finding
